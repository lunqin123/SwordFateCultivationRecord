# 「剑缘修仙录」全面分析报告

> 分析日期：2026-05-05

---

## 一、项目概览

修仙宗门模拟经营游戏，Godot 4.5.1 Mono (C#)，2D UI为主。玩家经营一个修仙宗门，招募弟子、建造设施、管理资源、发展道侣关系、探索秘境，从无名小宗成长为万宗至尊。

### 技术栈
- Godot 4.5.1 Mono + .NET SDK 9.0
- C# (file-scoped namespace)
- 全程序化UI（无.tscn布局文件）
- JSON序列化存档（System.Text.Json）

### 代码规模
| 类别 | 文件数 | 总行数(约) |
|------|--------|-----------|
| 核心系统 (Core) | 6 | ~1700 |
| 数据模型 (Models) | 9 | ~500 |
| 数据表 (Tables) | 7 | ~1400 |
| 子系统 (Systems) | 9 | ~1900 |
| UI | 6 | ~2800 |
| 配置/场景 | 5 | ~100 |

## 二、架构与框架结构

### 2.1 整体评价：良好

子系统划分清晰，GameManager 驱动每日流水线，EventBus 解耦各模块，静态数据表把内容与逻辑分离。整体架构合格，可维护性较好。

### 2.2 架构图

```
GameManager (Autoload 单例)
  ├── TimeSystem          (日历：年/月/日)
  ├── ResourceSystem      (7种资源：增删查+每日收入)
  ├── DiscipleSystem      (弟子管理：招募/任务/突破/离去)
  ├── FacilitySystem      (设施：建造/升级/每日产出)
  ├── EventSystem         (随机事件：15%日触发+权重+冷却)
  ├── CompanionSystem     (道侣：牵线/表白/双修/生子)
  ├── SectQuestSystem     (宗门任务：3并行/品级/刷新)
  ├── PlotSystem          (剧情：4章20阶段/条件驱动)
  ├── SecretRealmSystem   (秘境：3-5房间/8种类型/风险选择)
  ├── AchievementSystem   (成就：42个/7类别)
  ├── SaveLoadManager     (存档：10槽位/JSON)
  └── AllEquipment[]      (全局装备池)
```

### 2.3 每日流水线

```
1. FacilitySystem.ProcessDaily()     → 建造推进 + 收入计算
2. ResourceSystem.ApplyDailyIncome() → 资源到账
3. ProcessOuterDisciples()           → 外门产出/增长/晋升
4. CompanionSystem.ProcessDaily()    → 好感/双修/生子
5. DiscipleSystem.ProcessDaily()     → 任务执行/突破/体力/离去
6. AutoAssign (可选)                 → 自动分配任务
7. HandleNewborns()                  → 新生儿入宗
8. EventSystem.ProcessCooldowns()    → 冷却递减
9. 内门选拔倒计时                    → 7天倒计时
10. CheckSectLevelUp()               → 宗门晋升检查
11. Quests.CheckProgress()           → 任务进度
12. Plot.CheckProgress()             → 剧情进度
13. CheckAchievements()              → 成就检查
14. Time.AdvanceDay()                → 日期推进
15. Autosave (可选)                  → 自动存档
16. EventSystem.TryTriggerEvent()    → 15%随机事件
```

### 2.4 关键问题

#### 2.4.1 MainUI.cs 单体过大 (1804行)

一个文件包含了9个Tab页的全部构建、刷新逻辑、所有弹窗管理、装备UI、道侣UI、秘境UI、弟子详情面板等。没有遵循单一职责原则。

**建议**：拆分为多个 partial class：
```
MainUI.cs              → 主框架、初始化、Tab切换
MainUI.TopBar.cs       → 顶栏资源/时间显示
MainUI.OverviewTab.cs  → 总览Tab
MainUI.DisciplesTab.cs → 弟子Tab
MainUI.FacilitiesTab.cs→ 营造+灵筑Tab
MainUI.CompanionTab.cs → 道缘Tab
MainUI.QuestsTab.cs    → 门令Tab
MainUI.ArchiveTab.cs   → 记事+卷宗+剧情Tab
MainUI.Popups.cs       → 所有弹窗逻辑
MainUI.Equipment.cs    → 装备管理UI
```

#### 2.4.2 随机数种子不同步

每个子系统都独立创建 `new Random()`，存档中的 `RngSeed` 从未被写入或读取。同一存档每次加载后产生不同的随机结果，无法进行确定性回放或调试。

**建议**：GameManager 持有一个主 `Random`，子系统通过构造函数或方法参数接收。

#### 2.4.3 EventBus 重新订阅前未清理

`InitializeNewGame` 中调用 `EventBus.Clear()`，但 `LoadGame` 中在重新订阅事件前没有清理。如果在读档过程中有事件残留，可能导致重复触发。

## 三、存档系统缺陷

### 3.1 遗漏字段清单

| 遗漏字段 | 所在位置 | 影响 |
|----------|---------|------|
| `_herbAccum` | GameManager | 外门灵草小数累积归零 |
| `_oreAccum` | GameManager | 外门矿石小数累积归零 |
| `_tradeAccum` | GameManager | 外门经商小数累积归零 |
| `_outerPromoteAccum` | GameManager | 外门晋升内门进度归零 |
| `_exploreCount` | AchievementSystem | 秘境探索成就进度重置 |
| `AutoAssignEnabled` | GameManager | 自动分配开关状态丢失 |
| `RecruitTournamentDays` | GameManager | 内门选拔进度丢失 (LoadGame:767 强制=-1) |
| `RngSeed` | SaveGameData | 已声明但从未写入 |

> 注：`SaveGameData` 中 `HerbAccum`/`OreAccum`/`TradeAccum`/`OuterGrowthAccum` 属性已定义，但 `CreateSaveData` 未写入，`ApplySaveData` 未读取。

### 3.2 无存档损坏处理

`LoadFromSlot()` 在反序列化异常时返回 null，但 `LoadGame()` 没有事务性保护——加载到一半失败时游戏可能处于不一致状态。

### 3.3 无存档版本管理

`SaveVersion` 字段固定 "1.0"，没有版本校验逻辑。未来修改数据结构后，旧存档静默失败或产生异常数据。

## 四、游戏节奏与数值平衡

### 4.1 修炼进度 (以天赋50/悟性50/心情50的普通弟子为例)

| 阶段 | 所需进度 | 基础速度 | 修正后速度 | 预计天数 | 突破概率 | 突破灵石 |
|------|---------|---------|-----------|---------|---------|---------|
| 凡人 | — | — | — | — | — | — |
| →练气 | 180 | 6 | ~10/天 | ~18天 | ~65% | 50 |
| →筑基 | 350 | 5 | ~9/天 | ~39天 | ~55% | 150 |
| →金丹 | 700 | 4 | ~7/天 | ~100天 | ~38% | 500 |
| →元婴 | 1400 | 3 | ~5.5/天 | ~255天 | ~27% | 2000 |
| →化神 | 2500 | 2 | ~4/天 | ~625天 | ~17% | 6000 |
| →渡劫 | 4500 | 2 | ~4/天 | ~1125天 | ~12% | 18000 |
| →大乘 | 7000 | 1 | ~2/天 | ~3500天 | ~10% | 50000 |

> 修正速度 = 基础速度 × 天赋加成(1+天赋/200) × 心情加成(1+心情/200) × 悟性加成(1+悟性/400) × 灵根加成 × 装备加成 × 技能加成

**评价**：到金丹约需 1 游戏年，到元婴约需 2-3 游戏年。作为模拟经营游戏的核心长期目标，这个节奏合理。

### 4.2 外门弟子产出 (核心数值问题)

```
单个采集外门: 产 0.22灵草 + 0.16矿石/天, 消耗 0.25灵石/天
单个经商外门: 产 0.25灵石/天, 消耗 0.25灵石/天 → 净收益 0
```

**外门弟子对灵石的直接贡献几乎为零。** 其价值仅体现在：
- 资源多样性（灵草→炼丹、矿石→炼器）
- 闲置弟子 0.3%/天 概率晋升内门（极低）
- 声望驱动的被动增长（初始声望0时为零）

**问题**：花费50灵石招募外门，几乎看不到回报。玩家会感觉"花钱买了个寂寞"。

**建议**：经商外门净收益调至 +0.15~0.2 灵石/天，采集外门维护费减半或产出提升30%。

### 4.3 外门晋升内门速度

```
晋升概率 = 闲置人数 × 0.003/天
初始: 0闲置 × 0.003 = 0
有10外门(20%闲置=2人): 2 × 0.003 = 0.006/天 → 约167天晋升1人
```

**极其缓慢**。外门是内门弟子的主要来源之一，这个速度严重限制了内门扩张。

### 4.4 设施投资回报

| 设施 | 造价 | 日产 | 建造天数 | 回本周期(灵石换算) |
|------|------|------|---------|-------------------|
| 膳堂 | 60 | 10灵气/天 | 2 | ~6天 |
| 灵田 | 80 | 5灵草/天 | 3 | ~16天 |
| 静修室 | 100 | 5灵气/天 | 3 | ~20天 |
| 矿脉 | 120 | 4矿石/天 | 4 | ~30天 |
| 丹房 | 200 | 2丹药/天 | 5 | 需配合灵田 |
| 演武场 | 150 | 3灵气/天 | 4 | ~50天+战力加成 |
| 会客厅 | 200 | 2灵气/天 | 5 | 间接收益为主 |
| 藏经阁 | 300 | 1灵气/天 | 6 | 解锁自动分配 |
| 阵法殿 | 400 | 1灵气/天 | 8 | 高阶内容 |

早期最优建造顺序是固定的：**膳堂→灵田→静修室**，限制了策略多样性。

### 4.5 核心节奏问题

#### 早期游戏空洞

开局状态：1个内门弟子、0外门、500灵石。前20-30天：
- 建造1-2个设施（然后等待建造完成）
- 给唯一弟子分配任务
- 反复点击"下一天"
- 等7天内门选拔

操作密度极低，玩家大量时间在空等。

**建议**：
- 开局赠送3-5名外门弟子
- 首次建造时间减半（"宗门初创，人心振奋"）
- 第5天触发引导事件，介绍核心机制

#### 快进跳过事件 (严重设计缺陷)

```csharp
// GameManager:209
if (!isFastForward)
{
    PendingEvent = Events.TryTriggerEvent(SectLevel);
}
```

7天/30天快进时**完全跳过随机事件**。使用快进的玩家永久失去事件奖励，而手动逐天点击的玩家获得更多资源。效率工具变成惩罚工具。

**建议**：快进中正常触发事件，触发后自动暂停快进等待玩家选择。

### 4.6 弟子体力Bug

```csharp
// DiscipleSystem:166 — 阈值是 < 10
if (d.CurrentStamina < 10) { /* 强制休息 */ continue; }
// DiscipleSystem:182 — 任务消耗正好是10
d.CurrentStamina -= 10;
// DiscipleSystem:183 — 体力=10的弟子在此浪费一天
if (d.CurrentStamina <= 0) { d.Mood -= 1; continue; }
```

体力恰好为10的弟子：通过 `< 10` 检查 → 消耗10 → 变0 → 触发惩罚 → **当天无产出且心情-1**。

**建议**：将阈值改为 `<= 10` 或 `< 20`。

## 五、各系统深度分析

### 5.1 弟子系统

**优点**：
- 9种任务类型各有独立产出，与四维属性(天赋/悟性/体质/神识)关联明确
- 技能学习2%日概率+悟性加成，成长曲线合理
- 离去机制层次分明：低心情→低忠心→长期低心情→离去
- 任务熟练度(0-5级)提供+8%/级加成

**问题**：
- 没有弟子间社交互动（除道侣外）：无师徒、师兄弟、好友关系
- 任务没有"预期产出预览"，玩家不知道派弟子去做什么更划算
- `TaskTargetId` 字段已定义但从未使用（原设计可能是针对特定设施/弟子）
- `ProcessTeach` 给所有其他弟子加修炼进度，但没有上限或衰减

### 5.2 道侣系统

**优点**：
- 相性计算(境界+属性+年龄+忠心)公式合理
- 好感度日常变化：同任务+0.3/天，分离3天衰减0.1~0.3/天
- 双修加成5-50%（基于好感度），高境界带低境界有额外加成
- 自动求婚(好感≥75+2%日概率)、生子(已婚+好感≥70+0.3%日概率)
- 子嗣继承父母平均属性±10点随机浮动

**问题**：
- 随机配对2%/天对于小宗门（5-8人）触发频率极低
- 硬编码只允许异性道侣
- `CompanionSystem.RollChildSpiritRoot()` 与 `DiscipleSystem.RollSpiritRoot(true)` 概率完全相同但代码重复
- 子嗣从出生到成为战力之间有约14年的空白期，期间没有任何相关事件

### 5.3 秘境系统

**优点**：
- 8种房间类型(密室/兽穴/石碑/陷阱/灵泉/武库/修炼室/最终宝箱)
- 成功概率公式合理：`0.5 + (SectPower/100) / (dangerLevel + riskLevel)`
- 每房间2选1，风险与回报权衡明确

**问题**：
- 秘境入口触发条件不明确——需要从UI手动开启？
- `SectPower` 的增长路径不清晰（主要来自弟子守卫任务，数值偏小）
- 秘境探索结果对所有弟子生效（生命减少、修炼增长），缺乏针对性

### 5.4 事件系统

**优点**：
- 30+事件，覆盖机遇/危机/来客/内务/比试5类
- 权重随机+30天冷却，避免重复
- 事件结果同时影响资源/声望/战力/弟子属性/修炼进度
- 高阶事件随宗门等级解锁，保证后期有新内容

**问题**：
- `DiscipleStatEffects` 用数组 `[0]=忠心, [1]=心情, [2]=生命`，硬编码索引极易出错且不直观

```csharp
// EventData.cs 中的问题写法
DiscipleStatEffects = new[] { 5, 10, -10 }
// 5是什么？10是什么？-10是什么？必须查文档
```

**建议**：改为命名结构：
```csharp
public class DiscipleStatEffect
{
    public int LoyaltyChange { get; set; }
    public int MoodChange { get; set; }
    public int HealthChange { get; set; }
}
```

### 5.5 宗门任务系统

**优点**：
- 3个并行任务，完成后自动刷新
- 品级随宗门等级/设施提升(1-4品)，奖励递增
- 9种任务类型覆盖招募/建造/声望/资源/突破/升级/装备/外门/战力

**问题**：
- **任务只检查绝对当前值，不跟踪增量**。例如"积累200灵石"——如果玩家已有200灵石，接取瞬间即完成。应该记录接取时的基线。
- `CountProgress` 中 `GatherResource` 检查的是灵石而非资源总量，但任务描述说"灵石储量达到X"，合理但名字有误导性
- `UpgradeFacilities` 检查 `f.Level > 1` 而非升级次数

### 5.6 剧情系统

**优点**：
- 4章20阶段，条件驱动推进
- 支持自动触发和手动确认两种模式
- 阶段奖励包含资源和声望

**问题**：
- 如果剧情不承担"新手引导"功能，纯新玩家会迷失
- 剧情阶段的条件配置在 `PlotTable.cs`(330行)中，未能详细审查

## 八、未来方向建议

当前项目的核心循环（招募→培养→突破→经营）基础扎实。按优先级排列的扩展方向：

### 8.1 战斗/冒险系统
当前 `CombatPower` 只影响 `SectPower`，间接用于秘境成功率。没有实际战斗玩法。建议做一个简单的回合制战斗或自动战斗，让"训练"和"守卫"的价值更直观。

### 8.2 子嗣成长系统
子嗣已能出生但始终0岁。可加入：
- 子嗣按年成长（每年+1岁）
- 14岁后自动成为外门或内门弟子
- 道脉传承技能(id=10)在子嗣修炼中提供加成

### 8.3 外交/联盟系统
当前仅在事件文本中提及其他宗门。可扩展为：

## 九、总结

项目的**核心优势**：
- 架构清晰，子系统分工明确
- 游戏系统覆盖面广（修炼/经营/道侣/秘境/剧情/成就）
- 数值框架完整，可调参数丰富
- 事件内容丰富（30+个），中文本地化完整

项目的**核心短板**：
- 早期游戏节奏过慢，操作密度低
- 快进跳过事件是反直觉的设计缺陷
- 存档系统有多个遗漏字段
- UI代码过于集中，难以维护
- 外门弟子的投入产出比不合理

**总体评价**：作为一款修仙宗门模拟经营游戏，核心玩法循环已经可以运转，系统之间的耦合合理。当前最需要改善的是**早期游戏节奏**和**存档稳定性**，其次是对UI代码进行结构化拆分以支撑后续功能扩展。
