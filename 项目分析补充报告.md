# 「剑缘修仙录」分析补充报告

> 日期：2026-05-05 | 对第一轮分析的修正和补充

## 一、对第一轮分析的修正

### 1.1 新手引导：并非缺失

第一轮报告指出"没有新手引导"。**这不准确。** PlotTable 的前5个阶段就是精心设计的内置引导：

| 阶段 | 目标 | 引导内容 |
|------|------|---------|
| 序·祖师遗命 | 浏览宗门概况 | 让玩家熟悉界面布局 |
| 壹·广纳门徒 | 招募3名弟子 | 教会使用内门选拔 |
| 贰·安身立命 | 建造1座灵筑 | 引导到营造Tab |
| 叁·初窥门径 | 突破至练气期 | 教会分配修炼任务 |
| 肆·宗门初立 | 声望100+灵石500 | 全面发展概念 |

每个阶段都有 `Narrative`（叙事文本）、`Objective`（目标描述）、`CompletionHint`（具体操作提示）。**设计质量很高。** 问题在于：
- 剧情弹窗是否足够醒目，玩家会不会直接关掉
- 手动确认型(stage 1)需要玩家点击确认按钮，如果入口不明显可能卡住

### 1.2 快进跳过事件：补充说明

第一轮将此列为"严重设计缺陷"。更深层的分析是：设计意图可能是 **快进=安全的资源积累，正常速度=冒险的事件收益**，以此创造策略选择。但当前的问题在于事件奖励（声望/资源/修为加成）远大于快进省下的时间，导致玩家没有理由使用快进。这是一个**数值平衡问题**而非纯设计问题。

## 二、第一轮遗漏的重要发现

### 2.1 DiscipleStatEffect 类型不一致 (潜在编译问题)

`EventData.cs` 中定义了具名类：
```csharp
public class DiscipleStatEffect
{
    public int LoyaltyChange { get; set; }
    public int MoodChange { get; set; }
    public int HealthChange { get; set; }
}
public DiscipleStatEffect? DiscipleStatEffects { get; set; } // EventOutcome
```

但 `EventTable.cs` 中事件数据使用旧式数组语法：
```csharp
DiscipleStatEffects = new[] { 5, 10, -10 }  // 创建的是 int[]
```

同时 `EventSystem.cs:ApplyOutcome` 中以数组下标访问：
```csharp
d.Loyalty = Math.Clamp(d.Loyalty + outcome.DiscipleStatEffects[0], 0, 100);
d.Mood = Math.Clamp(d.Mood + outcome.DiscipleStatEffects[1], 0, 100);
```

**三者不一致。** 如果代码能编译通过，说明 DiscipleStatEffects 的实际类型是 `int[]` 而非 `DiscipleStatEffect?`，即 EventData.cs 中我看到的版本可能已被修改但 EventTable/EventSystem 尚未同步更新。需要确认并统一。

### 2.2 AchievementTable.TargetValue 虽定义但未使用

`AchievementTable` 中所有成就都定义了 `TargetValue`，但 `AchievementSystem.CheckCondition` 使用硬编码的 switch/if 语句检查每个成就，完全忽略了 `TargetValue`。

**影响**：如果未来要调整某些成就的门槛值（如"弟子达到25人"改为30人），需要在两个地方修改（表数据 + 检查逻辑），容易遗漏。

**建议**：将通用型成就（如弟子数、声望值、等级达标）改为数据驱动检查，仅保留特殊逻辑（如外门晋升、气运之子）的硬编码判断。

### 2.3 CultivationTable.IsMaxRealm 逻辑混乱

```csharp
public static bool IsMaxRealm(CultivationRealm realm, int layer)
{
    if (realm == CultivationRealm.GreatAscension) return true;
    return false; // 注释说"Mortal always promotes to QiRefining"
}
```

- `layer` 参数完全被忽略
- 注释描述的是 Mortal 而非整个函数的逻辑
- 如果将来在 GreatAscension 之后增加境界，需要修改此函数

**修正**：
```csharp
public static bool IsMaxRealm(CultivationRealm realm, int layer)
    => realm == CultivationRealm.GreatAscension && layer >= MaxRealmLayer(realm);
```

### 2.4 DiscipleNameTable 生成属性值之和超出预期

```csharp
int remaining = 200;
int t = rng.Next(20, 81);        // remaining -= t (不减去20的"基础值")
// ...
int co = rng.Next(20, min(81, remaining+20));
remaining -= (co - 20);          // 只减超出20的部分
```

由于每个属性前20点为"免费"不扣减 `remaining`，实际四个属性之和约为 **240-270**（而非直觉预期的200左右），平均每项60-67。

这对游戏性的影响：新手弟子属性起点较高，差异化空间缩小。如果希望弟子间有更大差异（有的特别聪明但体弱，有的体壮但悟性差），可以将 `remaining` 降至 120-140。

### 2.5 多个 Random 实例不一致

| 位置 | 随机数来源 |
|------|-----------|
| GameManager | `new Random()` |
| DiscipleSystem | `new Random()` |
| EventSystem | `new Random()` |
| CompanionSystem | `new Random()` |
| SecretRealmSystem | `new Random()` |
| SectQuestSystem | `new Random()` |
| DiscipleNameTable | `new Random()` |
| AudioSynth | `new Random(42)` (固定种子) |
| EquipmentTable | `Random.Shared` |
| MainUI | 部分使用 `new Random()` |

7个地方创建独立实例，1个用 `Random.Shared`，1个有固定种子。不统一，存档中的 `RngSeed` 字段毫无意义。**建议**：全部改为从 GameManager 注入。

### 2.6 GameSettings.LinearToDb 死代码

```csharp
// GameSettings:110
private static float LinearToDb(float linear) =>
    linear > 0.001f ? Mathf.LinearToDb(linear) : -80f;
```

此方法从未被调用。AudioManager 直接使用 `Mathf.LinearToDb()`。

### 2.7 EventTable 内容重复

| ID | 标题 | ID | 标题 | 重复点 |
|----|------|----|------|--------|
| 36 | 秘境入口(Lv.3) | 59 | 秘境现世(Lv.3) | 都是发现秘境，文本和选项高度相似 |

两个事件都是"发现秘境→探索/准备/放弃"的结构。建议合并或让59号有独特的叙事角度。

## 三、未充分评价但值得关注的系统

### 3.1 AudioSynth — 隐藏的亮点

`AudioSynth.cs` (346行) 是一个完整的**程序化音频合成引擎**：

- 4种音色生成器：纯音(GenTone)、钟声(GenBell)、铜锣(GenGong)、闪烁(GenShimmer)
- 噪声生成器带低通滤波
- 自定义包络(ADSR)：打击/钟鸣/反转/衰减 4种预设
- 多轨混音 + 归一化
- 16-bit WAV输出，手动构建RIFF头
- 30秒五声音阶(D宫调)冥想BGM，带交叉淡入淡出无缝循环

**这是项目中技术含量最高的代码**，却在上轮分析中完全没提。值得在项目文档中特别表彰。

### 3.2 SpriteSheetManager — 完善的图集系统

- 从JSON解析帧坐标和名称
- 提供资源图标/设施图标/头像的统一访问接口
- 已映射7种资源类型+10种设施类型+男女头像
- 空安全：任何查找失败都返回 null 而非崩溃

### 3.3 UITextures + 回退策略

设计了一个优雅的回退链：
```
尝试加载PNG纹理 → 成功则使用
                 → 失败则生成纯色2x2占位图
```

这意味着即使没有任何美术资源，游戏仍能运行（颜色编码的占位符）。与其他纯代码UI的风格一致。

### 3.4 SettingsWindow — 独立完整的设置系统

256行，3个Tab（画面/音频/游戏），功能齐全：
- 4档分辨率预设，全屏/垂直同步切换
- 主音量/音乐/音效独立滑块，BGM曲目选择
- 自动存档开关+间隔设置(3/5/7/10/15/20/30天)
- 存档管理（存录/开卷/归返山门按钮，在MainUI中传入回调）
- "恢复默认"按钮

**特别之处**：音频设置会即时生效（调用 `GameSettings.ApplyAudio()` 和 `GameSettings.ApplyDisplay()`），不需要重启。

## 四、数值补充分析

### 4.1 装备产出概率 (按宗门等级)

```
Lv.1: 凡品 100%
Lv.2: 凡品 80%, 灵品 20%
Lv.3: 凡品 60%, 灵品 30%, 宝品 10%
Lv.4: 凡品 44%, 灵品 30%, 宝品 20%, 仙品 6%
Lv.5: 凡品 30%, 灵品 30%, 宝品 25%, 仙品 15%
Lv.6: 凡品 20%, 灵品 30%, 宝品 30%, 仙品 20%
Lv.7: 凡品 10%, 灵品 30%, 宝品 30%, 仙品 30%
```

仙品在Lv.5以上开始稳定产出，品质提升速度合理。注意 `Random.Shared` 的使用在此处是线程安全的，因为Godot是单线程游戏循环。

### 4.2 宗门晋升声望阈值分析

| 等级 | 所需声望 | 增量 | 弟子上限 | 设施上限 |
|------|---------|------|---------|---------|
| Lv.1 | 0 | — | 5 | 2 |
| Lv.2 | 100 | 100 | 8 | 4 |
| Lv.3 | 300 | 200 | 12 | 6 |
| Lv.4 | 600 | 300 | 18 | 8 |
| Lv.5 | 1000 | 400 | 25 | 10 |
| Lv.6 | 1500 | 500 | 35 | 12 |
| Lv.7 | 2500 | 1000 | 50 | 14 |

Lv.6→Lv.7 是最陡峭的跨越（1000声望），这与剧情中"万宗至尊"的终极目标一致。但Lv.5→Lv.6只需要500，比Lv.4→Lv.5的400只多了100，梯度偏平。建议调至 2000。

### 4.3 声望获取途径汇总

| 途径 | 典型收益 | 频率 |
|------|---------|------|
| 随机事件(正向选择) | 5-60 | 约每7天一次 |
| 宗门任务完成 | 15-200 | 完成后刷新 |
| 剧情阶段完成 | 30-500 | 阶段性 |
| 成就解锁 | 10-400 | 一次性 |
| 弟子突破 | 间接(战力和后续产出) | 持续 |
| 年度总结 | 0 (仅展示) | 每年一次 |

早期声望主要靠剧情和宗门任务，中后期靠事件累积。**声望不会自然衰减**，这点合理——修仙宗门不应因不动而退步。

## 五、代码细节补充

### 5.1 SpriteSheetManager 初始化时机

```csharp
static SpriteSheetManager()
{
    LoadAtlas();  // 静态构造函数，首次访问时执行
}
```

静态构造函数在首次访问时执行。如果文件加载失败（如导出后路径不存在），`_frames` 为空字典但不会崩溃。所有 `GetIcon` 调用都检查 `_atlas == null || !_frames.TryGetValue`。**容错性好。**

### 5.2 AudioManager 导出容错

`ScanBGM()` 方法有两层回退：
1. `DirAccess.Open()` — 在导出中可能失败
2. 如果 BGM 列表为空，检查硬编码的已知 BGM 文件名

这个设计解决了 Godot 导出后 `DirAccess` 不可用的经典问题。

### 5.3 BGM 自动循环

```csharp
_bgmPlayer.Finished += () =>
{
    int next = (GameSettings.BgmIndex + 1) % _bgmPaths.Count;
    GameSettings.BgmIndex = next;
    PlayBgmTrack(next);
};
```

曲目播放完毕后自动跳到下一首。但 `BgmIndex` 被更新后，下一次手动切歌时会从新位置开始而非用户之前选的位置。如果用户想固定听某一首，当前设计不支持循环单曲。

**建议**：在 SettingsWindow 中增加循环模式选项（单曲循环/列表循环/随机）。

### 5.4 图集JSON格式

`SpriteSheetManager` 的 JSON 解析假设 `frame` 对象有 `x/y/w/h` 字段。这与 Aseprite 导出的 sprite sheet JSON 格式一致——说明开发流程中使用了 Aseprite 做精灵图编辑，有 `addons/AsepriteWizard` 插件辅助。这是一个完整的资产管线。

## 七、总评修正

第一轮报告的总体评价基本准确，但需修正以下几点：

1. **"缺少新手引导"是误判** — PlotTable 的内置引导设计得很好，撤回此批评
2. **代码质量比初评更好** — AudioSynth、SpriteSheetManager、UITextures 回退链、SettingsWindow 等都展现了高质量的防御性编程和完整的功能设计
3. **项目完成度比预期高** — 不仅有核心玩法循环，音效系统、设置系统、精灵图集管线都是完整的

修正后总体评价：**这是一款核心玩法扎实、支持系统完善、有完整内容管线的修仙模拟经营游戏。当前最需要改善的是：存档数据完整性(stage 1中列出的5+遗漏字段)、代码一致性(DiscipleStatEffect类型统一/Random实例统一)、以及数值微调(外门产出/早期节奏)。**
